我那些前后端分离的经历

发表于 2016-06-08   |   分类于 前端   |     |  

前言

工作也有一段时间了，在前端这一块也摸索了许久，至于多多少少有点感想和看法，对于前端的发展和模式也接触了不少，而至于前端的工作流程也取决于公司技术框架结构。
最值得探讨的不过是模板引擎和数据结构，其实还有工具化和工程化，因为它会决定公司的技术框架和工作流程，甚至方向。
在web应用天下，有两头狮子，前端攻城狮和后端攻城狮；它俩结合干了很多不为人知的羞羞事情。老是将数据这块封装来封装去，搞得我赶紧写篇博客压压惊。

渲染，你在哪

要前后端分离，一切取决于数据模板在哪里渲染。

HTML模板引擎是将数据填充进指定的HTML模板去，按模板定制的样式把数据生成了最终需要的业务数据模型（比如网页）。

模板引擎不属于特定技术领域，它是跨领域跨平台的概念。在Asp下有模板引擎，在PHP下也有模板引擎，在C#下也有，甚JavaScript、WinForm开发都会用到模板引擎技术。

目前我将它归纳为三种渲染的层面：

  • 后端来处理，在服务器产出HTML推回给浏览器。
  • 前端来处理，对服务器请求数据，进而渲染页面。
  • 借助Node来处理，综合阶段一、二的混合物。
后端来渲染数据模板

后端来处理模板引擎，经典 jsp、php、asp 就是这样弄的；也许这会被看成早期或者比较传统的渲染处理方式。

浏览器发起请求到服务器，服务器接到请求做了相应处理，把数据注入模板引擎，将生成的HTML返回给浏览器。

相对来说，浏览器不用耗多大力气来处理渲染，除去了重绘和计算的麻烦，也对SEO很友好。

而如果小面积需更新，服务器会将整个页面返回到浏览器重新绘制了；一个页面切分了多个模板，在后台处理整合输出，这不算是高效的做法。再然后一个页面上多个交互产生多个状态界面，那数据逻辑拆分和定位也是零零碎碎的，难以维护。加上业务逻辑不断扩大，模板引擎会越来越复杂，固然会不断接近崩溃的边缘。

基本总结为几点：

  • 后期维护性差
  • HTML模板不断膨胀
  • 前端专业化程度低
  • 需求响应转换速度慢
  • 后期交互优化困难

目前很多这样的框架都引进了ajax的技术，在一些交互上做异步请求，局部更新的问题也就解决了，这里要考虑几点，局部更新了会不会连带其它地方有所更新，页面状态定位，慢慢的越来越多JavaScript代码。再后来交互也丰富起来了，问题慢慢地难以处理，或者绝大部分开始转移ajax技术来处理页面数据。

ajax有很多优点

1、最大的一点是页面无刷新，用户的体验非常好。
2、使用异步方式与服务器通信，具有更加迅速的响应能力。
3、可以把以前一些服务器负担的工作转嫁到客户端，利用客户端闲置的能力来处理，减轻服务器和带宽的负担，节约空间和宽带租用成本。并且减轻服务器的负担，ajax的原则是“按需取数据”，可以最大程度的减少冗余请求，和响应对服务器造成的负担。
4、基于标准化的并被广泛支持的技术，不需要下载插件或者小程序。

ajax的缺点

1、ajax不支持浏览器back按钮。（有人去模拟触发浏览器back状态）
2、安全问题 AJAX暴露了与服务器交互的细节。（开始考虑数据安全）
3、对搜索引擎的支持比较弱。（SEO是个痛点，来老板说要弄两套）
4、破坏了程序的异常机制。（像Ajax.dll，Ajaxpro.dll这些）
5、不容易调试。（复杂）

前端来渲染数据模板

市面产出了许多前端模板引擎，使用JavaScript ajax请求数据来封装HTML，做逻辑处理产出HTML；对SEO不友好，前端要弄很多业务逻辑，性能不一定好，安全需要考虑，交互丰富起来嘿嘿嘿。前端要求高了，代码没少你。

前端终于可以当家做主了，前端攻城狮终于站起来啦。看，他们威风迎面而来。

这里列举一些模板引擎：

名称 说明
jade jade是超高性能的node JavaScript模板引擎，有着非常强大的API和大量杰出的特性。它主要针对node的服务端。
swig A simple, powerful, and extendable JavaScript Template Engine。
mustache Mustache是logic-less模板语法，可以使用在HTML，配置文件，源代码等等地方。它是使用哈希表或者对象提供的值来扩展模板标签。
template template.js 一款 JavaScript 模板引擎，简单，好用。提供一套模板语法，用户可以写一个模板区块，每次根据传入的数据，生成对应数据产生的HTML片段，渲染不同的效果。
ejs 　　EJS会使用客户端模板从JavaScript中清理HTML代码，整理好这些代码之后，JavaScript代码就会变得更加整洁和有条理。
Handlebarsjs Handlebars给构建语义模板提供强大的基础支持。
Transparency 　Transparency是一个强大的客户端网站模板引擎，可直接绑定数据到DOM，包括一些很棒的特性如HTML模板、JS视图逻辑。支持IE9+, Chrome, Firefox, iOS, Android, Etc.

必须提到数据接口，而不得不提Restful接口，REST指的是一组架构约束条件和原则。 如果一个架构符合REST的约束条件和原则，我们就称它为RESTful架构。
而资源与URI，在获取资源，RESTful架构遵循统一接口原则，统一接口包含了一组受限的预定义的操作，不论什么样的资源，都是通过使用相同的接口进行资源的访问。接口应该使用标准的HTTP方法如GET，PUT和POST，并遵循这些方法的语义。
web基本技术：URI（统一资源标示符）HTTP（超文本传输协议）（post、get、put、delete）Hypertext。

  • 采用URI标识资源
  • 使用“链接”关联相关的资源
  • 使用统一的接口
  • 使用标准的HTTP方法
  • 支持多种资源表示方式
  • 无状态性

我们聊聊前端一些接入模板引擎的事情，简简单单就是一些H5活动，或者小页面会自己封装数据，定位HTML模板；写个资源控制；这个没什么好讲的。
看经典的backbone框架，因为它是结合underscore基础库，这个underscore有自己的模板引擎，用过它的人都知道。而我们来看看最新的一些框架angular、Vue、React。在最新MVVM模式里，模板引擎被提升了，它不只是简单模板呈现，或多或少添加许多指令，做了很多数据动态的工作。
angularjs本身是很好的前端模板引擎，未来发展就是后端的mvc产生json视图（view）作为前端的模型（model），而整个前端的mvc是后端的视图（view），中间通信就靠json。这样前后端高度解耦，可以完全达到模块化设计的要求。

到了React，就是就组件化了。（暂不提，后续更新）
